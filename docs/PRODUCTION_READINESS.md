# Production Readiness

MorphoJet is not production-grade until this checklist is satisfied by current code, current tests, and current release artifacts.

## Release Contract

Production-grade means:

- A tagged release can be built from CI.
- macOS and Linux binaries are published with checksums.
- The CLI exits non-zero on invalid inputs and never silently overwrites existing measurement outputs.
- Output CSV files are complete for the successful run or absent for a failed run.
- Every public compatibility claim is tied to a parity report.
- Every performance claim is tied to a benchmark report.

## Claim Policy

Until every required gate is complete, documentation may claim the narrow L3 benchmark result and the supported-column handoff preflight, but must not say "production-ready" or "replaces CellProfiler workflows" without an external workflow trial report and evidence package accepted by release gate.

The release-gate report's top-level `production_claim_status` is the machine-readable summary for this policy. It is expected to remain `INCOMPLETE` until clean-git, L3 provenance, external L4 workflow, external L4 evidence package, saved external L4 reviewer reports, stable GitHub release, and saved stable-release verifier checks are all present and passing in the same report. Reports also include top-level `claim_status`, `evidence_scope`, and `final_production_signoff`: every non-final report must remain `NOT_PRODUCTION_CLAIM`, `RELEASE_GATE_PRECHECK`, and `false`; a final production report must be `FINAL_PRODUCTION_CLAIM`, `FINAL_PRODUCTION_RELEASE_GATE`, and `true`. Reports also include top-level `missing_or_failed_checks`, which names the exact production-claim checks still blocking a production-ready or CellProfiler-replacement claim; JSON reports additionally include `production_claim_checklist`, and Markdown reports render that same checklist so every audit check maps to required evidence and next reviewer action for human signoff. Review saved reports with `benchmark/verify_release_gate_report.py --verify-git-commit --expect-missing-checks ...` to require that the recorded commit is reachable, `metadata.generated_at_utc` is UTC, the blocker list is exactly the intended list for that milestone, and the top-level claim-scope labels match the final-signoff state; final production reports should use `--require-clean-git-metadata --verify-git-commit --expect-missing-checks none`. Final production/stable-release validation should run release gate with `--require-production-claim`; that mode fails the overall report unless the production-claim audit is `PASS`. Saved final reports should also pass `benchmark/verify_release_gate_report.py --require-report-pass --require-clean-git-metadata --verify-git-commit --require-production-claim-pass --expect-missing-checks none`, which rejects production PASS reports with duplicate gate names, missing required production gate entries or final production metadata flags, missing or tampered production checklist rows or final claim-scope labels, true boolean metadata flags without matching gates, relative or missing external paths, external metadata paths without validation gates, external trial reports that lack a bound pre-execution READY readiness summary, missing saved reviewer/verifier report metadata, relative path-valued `metadata.argv` entries for external evidence, reviewer reports, or explicit report outputs, recorded `--out-json` values that do not match the report under review, live GitHub release gate commands that do not match the recorded tag/kind/output JSON and production repo, saved reviewer metadata paths without matching gates, saved reviewer gate commands that omit file rechecks, PASS requirements, package source-trial binding, stable release checks, git commit verification, or expected tag binding, stable release identity, clean-git metadata, a reachable report commit, UTC report metadata, or consistency between top-level status, recorded gate statuses, and recorded production-claim enforcement metadata. `release_gate.py` writes those production evidence path fields, argv path values, live GitHub verifier JSON paths, and saved reviewer gate command report paths as absolute paths so saved reports remain auditable outside the original shell working directory.
